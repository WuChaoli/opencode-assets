# Canvas 驱动开发完整工作流

## 15 步工作流

### 第一阶段：准备（Step 1-2）

#### Step 1: 理解核心理念

```
Canvas 白板作为唯一真相源，代码是其序列化形式
图形语言优于文字描述
人类负责架构设计，AI 负责代码实现
```

**核心转变**：
- 传统：代码 → 口头沟通 → 脑补架构 → 代码失控
- Canvas：代码 ⇄ 白板 ⇄ AI ⇄ 人类

#### Step 2: 准备工具环境

**必需工具**：
- [Obsidian](https://obsidian.md/) - 免费开源白板工具
- AI 助手（支持读取 Canvas JSON 格式）
- 目标项目代码库

**可选工具**：
- VS Code + Mermaid 预览插件
- Git（版本控制 .canvas 文件）

---

### 第二阶段：生成白板（Step 3-5）

#### Step 3: 生成初始架构白板

**操作**：
1. 向 AI 提供项目代码路径
2. 使用 AI 架构总师提示词（见 `ai-architect-prompt.md`）
3. AI 自动生成 `.canvas` 文件

**示例指令**：
```
分析 /path/to/project 的架构，生成 Canvas 白板。
粒度：文件级
关注：依赖关系、数据流、潜在风险
```

#### Step 4: 用 Obsidian 打开 .canvas 文件

**检查项**：
- [ ] 模块识别是否正确
- [ ] 文件/API 调用关系是否准确
- [ ] 关键依赖连线是否完整

#### Step 5: 人工优化白板架构

**优化操作**：
1. 拖动调整模块位置使布局清晰
2. 补充 AI 遗漏的隐式依赖连线
3. 添加注释节点标注关键设计决策
4. 删除冗余或错误的连接
5. 应用颜色编码（参见 `architecture-spec` Skill）

---

### 第三阶段：建立同步机制（Step 6）

#### Step 6: 建立代码-白板同步机制

**同步规则**：
| 代码变更 | 白板操作 |
|:---|:---|
| 新增文件 | 添加新节点 |
| 新增 import | 添加新连线 |
| 删除文件 | 删除对应节点 |
| 重命名 | 更新节点标签 |
| 重构 | 调整连线关系 |

**触发时机**：
- 每次 PR/MR 合并前
- 重大功能完成后
- 定期检查（建议每周）

---

### 第四阶段：白板驱动开发（Step 7-9）

#### Step 7: 白板驱动 AI 编程（新功能开发）

**工作流**：
1. 在白板上画出新模块框
2. 画出预期调用关系（连线）
3. 添加节点描述（职责、接口）
4. 导出白板 JSON 发送给 AI
5. 指令："按照这个架构图实现具体代码"

**示例**：
```
我在白板上画了一个 PaymentService 模块，
连接到 OrderService 和 Database。

请根据白板实现代码：
- PaymentService 负责支付处理
- 调用 OrderService.update_status()
- 写入 Database.payments 表
```

#### Step 8: 白板驱动代码重构

**工作流**：
1. 在白板上删除/重连模块间的依赖线
2. 标注需要拆分的大模块
3. 发送修改后的白板给 AI
4. 指令："按新架构重构代码，列出需要修改的文件清单"

**示例**：
```
我在白板上将 payment_service 拆分为：
- PaymentProcessor（处理支付逻辑）
- PaymentValidator（验证支付信息）

请按新架构重构代码。
```

#### Step 9: 白板辅助 Code Review

**工作流**：
1. Review 前先看白板全局架构
2. 识别异常连线：
   - 前端直接连数据库？
   - 循环依赖？
   - 跨层调用？
3. 在白板上标注问题点
4. 讨论时指着白板说明

**检查清单**：
- [ ] 是否有绕过 API 层的直接调用？
- [ ] 是否存在循环依赖？
- [ ] 新增模块是否符合现有架构？
- [ ] 数据流是否合理？

---

### 第五阶段：团队协作（Step 10-11）

#### Step 10: 用白板加速团队协作

**场景应用**：
| 场景 | 使用方式 |
|:---|:---|
| 新人入职 | 1 分钟看白板理解全局 |
| 需求评审 | 在白板上画出变更范围 |
| 技术方案会议 | 投屏白板而非代码 |
| 会后分工 | 将白板标注转化为任务 |

#### Step 11: 维护白板与代码一致性

**一致性检查**：
1. 每次 PR/MR 合并前检查白板是否需要更新
2. 定期运行校验：对比白板 JSON 与实际代码依赖
3. 发现不一致时**优先修正白板**（因为白板是真相源）

**校验脚本思路**：
```python
# 伪代码
actual_deps = extract_dependencies_from_code()
canvas_deps = parse_canvas_json()
diff = actual_deps - canvas_deps
if diff:
    print(f"白板缺失依赖: {diff}")
```

---

### 第六阶段：扩展应用（Step 12-15）

#### Step 12: 扩展应用场景

| 场景 | 应用方式 |
|:---|:---|
| 接手遗留项目 | 自动生成白板快速理解 |
| 性能优化 | 用白板标注热点路径 |
| 安全审计 | 检查白板上的敏感数据流向 |
| API 设计 | 画出服务间调用拓扑 |

#### Step 13: 明确项目类型

根据项目类型调整白板策略：
- **单体应用**：文件级粒度，关注模块边界
- **微服务架构**：服务级粒度，关注 RPC 通信
- **前后端分离**：分层展示，关注 API 契约

#### Step 14: 选择白板粒度级别

| 粒度 | 适用场景 |
|:---|:---|
| A-文件级 | 新手、小项目（推荐入门） |
| B-类/函数级 | 复杂业务模块分析 |
| C-服务级 | 微服务、大型项目 |

#### Step 15: 持续迭代工作流

**定期回顾**：
- [ ] 白板是否反映真实架构？
- [ ] 团队是否在使用白板？
- [ ] 节点命名和布局规则是否需要优化？

**进阶探索**：
- 白板与 CI/CD 集成（PR 触发白板 diff 检查）
- 分享最佳实践案例到团队知识库
- 探索多白板管理（按领域/模块拆分）
