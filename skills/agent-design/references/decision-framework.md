# 决策框架：Agent 设计核心原则

## 1. Conway's Law for Agents

> Agent 的切分方式应该反映任务的真实依赖结构。

### 核心思想

Melvin Conway 提出的原始法则："设计系统的组织，其产生的设计等同于组织之间的沟通结构。"

应用到 Agent 设计：
- 如果任务之间**紧密协作**（共享上下文多）→ 同一个 Agent
- 如果任务之间**松散耦合**（独立性强）→ 不同 Agent

### 实际应用

**案例：Android 项目**
```
UI 开发          网络层开发         数据层开发
   ↓                ↓                ↓
共享上下文少       共享上下文少       共享上下文少
工具链差异大       工具链差异大       工具链差异大
→ 分成 3 个 Agent → 场景切分
```

**案例：代码审查流程**
```
安全审查         性能审查           可读性审查
   ↓                ↓                ↓
共享被审查代码    共享被审查代码     共享被审查代码
工具集完全相同   工具集完全相同     工具集完全相同
→ 合并为 1 个 Agent（或按严重程度区分）
```

## 2. Context Affinity（上下文亲和度）

> 将高上下文亲和度的操作放在同一个 Agent 中。

### 定义

上下文亲和度 = 两个任务共享的上下文量 / 总上下文量

- **亲和度 > 70%** → 强关联，放在同一个 Agent
- **亲和度 30-70%** → 中度关联，可拆分但需协调
- **亲和度 < 30%** → 弱关联，适合独立成 Agent

### 计算示例

**测试开发场景：**
```
任务 A：编写测试代码
  - 需要了解：被测代码结构、测试框架 API、Mock 策略
  
任务 B：运行测试并分析结果
  - 需要了解：被测代码结构、测试框架输出、错误模式

共享上下文：被测代码结构（60% 的上下文）
总上下文：代码结构 + API + Mock + 输出 + 错误模式
亲和度 ≈ 60% → 中度关联，但端到端更适合
```

**推荐**：合并为 `test-engineer` Agent（端到端）

### 反模式

❌ **过度拆分**
```
test-write → test-run → test-analyze → test-report
（4 个 Agent，每个只做一件事）
问题：上下文在 4 个 Agent 之间反复传递，协调成本高
```

✅ **适度合并**
```
test-engineer（端到端）
- 写测试、运行、分析、报告一体化
- 保持上下文连续
```

## 3. Single Responsibility + Composability

### Single Responsibility

每个 Agent 应该：
1. 有明确的**输入**：任务描述 + 上下文范围
2. 有明确的**输出**：可交付的单一结果
3. 有明确的**工具集**：最少必要工具（2-5 个）

### Composability

Agent 应该可以被组合使用：
```
Orchestrator
  ├─ Subagent A → 输出摘要 A
  ├─ Subagent B → 输出摘要 B
  └─ Subagent C → 输出摘要 C
       ↓
   综合结果 → 主 Agent 决策
```

### 验证标准

问自己 3 个问题：
1. **如果去掉这个 Agent，主 Agent 需要多做多少工作？**
   - > 2x → 保留（价值高）
   - 1-2x → 考虑合并
   - < 1x → 删除（价值低）

2. **这个 Agent 的职责能用 1 句话描述清楚吗？**
   - 能 → 保留（职责清晰）
   - 不能 → 拆分或重构

3. **这个 Agent 和其他 Agent 共享完全相同的工具集吗？**
   - 否 → 保留（差异化能力）
   - 是 → 考虑合并（功能冗余）

## 4. 权限最小化原则

### 核心规则

**只给 Agent 必要的工具权限。**

### Subagent 特殊规则

OpenCode 规定：Subagent 的 `permission` **只能使用 `allow` 或 `deny`**，禁止使用 `ask`。

原因：Subagent 在后台执行，使用 `ask` 会阻塞等待用户确认。

### 推荐配置

| Agent 类型 | edit | bash | webfetch |
|-----------|------|------|----------|
| 只读审查 | deny | deny | allow |
| 规划分析 | deny | deny | allow |
| 文档编写 | allow | deny | allow |
| 完整开发 | allow | allow | allow |

### Tools vs Permission 详解

| 字段 | 层级 | 作用 | 值类型 |
|------|------|------|--------|
| `tools` | 上游 | 控制工具是否可用 | boolean |
| `permission` | 下游 | 控制工具如何执行 | ask/allow/deny |

**使用原则**：
- `tools: false` → 工具完全不可见
- `tools: true + permission: deny` → 可见但被拒绝
- `tools: true + permission: ask` → 可见但需确认（仅 Primary Agent）
- `tools: true + permission: allow` → 可见且自动执行

## 5. Token 经济性

### 现实检查

Multi-agent 系统使用约 **15x 更多 tokens** 单 Agent 聊天。

**计算：**
- 简单聊天：~2K tokens
- 单 Agent：~8K tokens（4x）
- Multi-agent：~30K tokens（15x）

### 何时值得投入？

✅ **值得的情况**：
- 任务需要并行探索（多数据源）
- 上下文污染风险高（大代码库）
- 质量要求高（研究、架构设计）

❌ **不值得的情况**：
- 简单任务（1-3 文件修改）
- 高步骤间依赖（无法并行）
- 成本敏感（token 预算有限）

### 优化策略

1. **模型路由**：Haiku（快速/便宜）→ Sonnet（平衡）→ Opus（深度）
2. **任务粒度**：Agent 只做信息收集，不做实现
3. **上下文压缩**：使用 subagent 隔离污染

## 6. 设计决策树

```
开始设计 Agent 体系
    │
    ├─ 任务复杂度评估
    │   ├─ 简单（1-3 文件）→ 单 Agent
    │   └─ 复杂（多模块/多阶段）→ 继续
    │
    ├─ 上下文特征分析
    │   ├─ 任务间共享上下文 > 50%？→ 场景切分
    │   └─ 任务间共享上下文 < 50%？→ 功能切分
    │
    ├─ 工具能力差异
    │   ├─ 工具集完全相同？→ 考虑合并
    │   └─ 工具集差异大？→ 保持分离
    │
    └─ 并行化需求
        ├─ 需要并行执行？→ 功能切分（独立上下文）
        └─ 顺序依赖强？→ 场景切分（减少协调成本）
```

## 7. 检查清单（最终验证）

### 创建前检查

- [ ] 明确项目类型和技术栈
- [ ] 评估任务复杂度（简单/中等/复杂）
- [ ] 识别当前痛点（上下文污染？协调困难？）
- [ ] 确认团队规模和协作需求

### 设计时检查

- [ ] 每个 Agent 有明确的输入/输出/工具集
- [ ] Agent 职责能用 1 句话描述清楚
- [ ] 工具权限遵循最小化原则
- [ ] Subagent 不使用 `permission: ask`

### 创建后检查

- [ ] 验证 Agent 可被正确触发（description 包含触发词）
- [ ] 测试 Agent 在孤立环境下能独立完成任务
- [ ] 检查是否存在工具集完全相同的冗余 Agent
- [ ] 评估 token 使用是否在可接受范围
