# 评估指南：如何评估 Agent 设计质量

## 评估维度

### 1. 职责清晰度（Clarity of Responsibility）

**评估标准**：
- [ ] Agent 职责能用 1 句话描述清楚
- [ ] 职责数量不超过 5 条
- [ ] 与其他 Agent 职责无重叠

**评分**：
- 5 分：职责单一且清晰，无歧义
- 3 分：职责较清晰，但有轻微重叠
- 1 分：职责模糊或过于宽泛

**示例**：
```yaml
# 5 分
description: "代码安全审查专家：检查 SQL 注入、XSS 等漏洞。触发条件：代码审查、PR review、安全检查时。"

# 1 分
description: "帮助开发者写更好的代码"
```

---

### 2. 工具集合理性（Tool Appropriateness）

**评估标准**：
- [ ] 工具数量 2-5 个（最少必要集）
- [ ] 工具与职责匹配
- [ ] 无冗余工具
- [ ] 权限最小化

**评分**：
- 5 分：工具集精准，权限最小化
- 3 分：工具集合理，但有 1-2 个多余工具
- 1 分：工具集过多或与职责不匹配

**示例**：
```yaml
# 5 分（审查 agent）
tools: [Read, Grep, Glob]
permission:
  read: allow
  edit: deny
  bash: deny

# 1 分（审查 agent 有写权限）
tools: [Read, Write, Edit, Bash, Grep, Glob, WebSearch]
permission:
  edit: allow
  bash: allow
```

---

### 3. 触发可靠性（Trigger Reliability）

**评估标准**：
- [ ] Description 包含明确触发条件
- [ ] 触发关键词具体可判定
- [ ] 不会误触发或漏触发

**评分**：
- 5 分：触发条件明确，100% 可判定
- 3 分：触发条件较明确，偶有歧义
- 1 分：触发条件模糊，依赖猜测

**示例**：
```yaml
# 5 分
description: "代码安全审查专家：检查 SQL 注入、XSS 等漏洞。触发条件：代码审查、PR review、安全检查时。"

# 1 分
description: "当你需要帮助时使用此 agent"
```

---

### 4. 上下文隔离度（Context Isolation）

**评估标准**：
- [ ] 探索型任务委托给 subagent
- [ ] 主 agent 上下文保持干净
- [ ] Subagent 之间上下文独立

**评分**：
- 5 分：完美隔离，无上下文污染
- 3 分：基本隔离，偶有泄漏
- 1 分：严重污染，上下文混乱

**检查方法**：
```
查看主 agent 历史：
- 是否有大量探索性操作？（应委托给 explore agent）
- 是否混杂不同任务的历史？（应拆分）
```

---

### 5. 协作效率（Collaboration Efficiency）

**评估标准**：
- [ ] Agent 间协调次数合理（< 5 次/任务）
- [ ] 无重复工作
- [ ] 结果传递高效

**评分**：
- 5 分：协调高效，无冗余
- 3 分：基本高效，偶有冗余
- 1 分：协调混乱，大量冗余

**计算方法**：
```
协调效率 = 实际产出 / (Agent 调用次数 × 平均 token 消耗)

目标：> 0.7（每 token 有 70% 产出）
```

---

### 6. 可维护性（Maintainability）

**评估标准**：
- [ ] Agent 数量合理（3-7 个）
- [ ] 命名规范
- [ ] 文档完整
- [ ] 易于扩展

**评分**：
- 5 分：结构清晰，易于维护
- 3 分：基本可维护，有小问题
- 1 分：难以维护，需要重构

**检查清单**：
- [ ] 新增 agent 是否需要修改多个文件？
- [ ] 修改一个 agent 是否影响其他 agent？
- [ ] 是否有 agent 文档说明设计决策？

---

## 综合评分

### 评分公式

```
总分 = 职责清晰度 × 0.25 +
       工具集合理性 × 0.20 +
       触发可靠性 × 0.20 +
       上下文隔离度 × 0.15 +
       协作效率 × 0.10 +
       可维护性 × 0.10

满分：5 分
优秀：> 4.0 分
良好：3.0-4.0 分
及格：2.0-3.0 分
不及格：< 2.0 分
```

### 评分示例

**案例：Android 项目 Agent 体系**

| 维度 | 得分 | 权重 | 加权得分 |
|------|------|------|---------|
| 职责清晰度 | 5 | 0.25 | 1.25 |
| 工具集合理性 | 4 | 0.20 | 0.80 |
| 触发可靠性 | 5 | 0.20 | 1.00 |
| 上下文隔离度 | 4 | 0.15 | 0.60 |
| 协作效率 | 4 | 0.10 | 0.40 |
| 可维护性 | 4 | 0.10 | 0.40 |
| **总分** | | | **4.45** |

**评级**：优秀（> 4.0）

## 评估流程

### Step 1: 静态评估

检查 Agent 配置文件：
1. 读取所有 `.md` agent 文件
2. 检查 frontmatter（description、tools、permission）
3. 检查职责定义（Responsibilities）
4. 检查约束条件（Constraints）

### Step 2: 动态评估

运行测试任务：
1. 执行典型任务，观察 Agent 调用链
2. 统计调用次数和 token 消耗
3. 检查上下文污染情况
4. 验证结果质量

### Step 3: 综合评分

根据静态和动态评估，计算综合得分。

### Step 4: 生成报告

输出评估报告：
```
Agent 体系评估报告
==================

总体评分：4.2/5.0（良好）

详细评分：
- 职责清晰度：5/5 ✅
- 工具集合理性：4/5 ✅
- 触发可靠性：5/5 ✅
- 上下文隔离度：3/5 ⚠️
- 协作效率：4/5 ✅
- 可维护性：4/5 ✅

改进建议：
1. 上下文隔离度有待提升：explore 任务未完全委托给 subagent
2. 建议将 explore 阶段独立为 subagent 调用

优化后预期评分：4.6/5.0
```

## 持续改进

### 监控指标

1. **Token 效率**：每任务平均 token 消耗
2. **调用频率**：每个 agent 被调用次数
3. **成功率**：任务完成率
4. **用户满意度**：用户反馈评分

### 迭代优化

```
每月 Review：
1. 检查监控指标趋势
2. 识别瓶颈 agent（调用过多/过少）
3. 收集用户反馈
4. 调整 agent 边界或新增 agent
5. 重新评估体系质量
```
