# 重构模式

## 模式 1: 内容精简

**症状**: SKILL.md 超过 200 行

**修复**:
1. 识别哪些内容可以移到 references/
2. 在 SKILL.md 中保留导航链接
3. 创建或更新 references/index.md

**示例**:
```
# Before (SKILL.md 350 行)
## API Reference
[200 行 API 文档]

# After (SKILL.md 150 行)
## References
- `references/api.md` - API 文档
```

## 模式 2: 触发优化

**症状**: Skill 有时触发有时不触发

**修复**:
1. 检查 description 是否包含具体关键词
2. 添加更多触发场景到 When to Use
3. 确保关键词足够独特

**示例**:
```yaml
# Before
description: "帮助处理数据库"

# After
description: "PostgreSQL 数据库操作指南。触发条件：SQL 查询、数据库优化、表设计时。"
```

## 模式 3: 指令冲突消除

**症状**: Skill 输出不一致或忽略某些指令

**修复**:
1. 找出矛盾的指令
2. 统一为一个明确的指令
3. 如果有条件差异，用 if-else 明确说明

**示例**:
```markdown
# Before (冲突)
- 使用英式英语
- 使用美式英语

# After (统一)
- 使用英式英语（UI 字符串除外，用美式英语）
```

## 模式 4: 示例补充

**症状**: 示例不可复现或不足 3 个

**修复**:
1. 补充缺失的前提条件
2. 添加环境/依赖说明
3. 实际运行验证

**示例**:
```markdown
### Example 1: [场景]

**前提**: [环境要求]
**输入**: [起始条件]
**步骤**:
1. [步骤 1]
2. [步骤 2]
**验收**: [如何判断成功]
```

## 模式 5: 范围拆分

**症状**: Skill 做了太多事

**修复**:
1. 按工作流阶段拆分
2. 创建多个聚焦的 Skill
3. 在 references 中建立关联

**示例**:
```
# Before
database-skill/ (涵盖查询、设计、优化、迁移)

# After
database-query/ (查询操作)
database-design/ (表设计)
database-optimize/ (性能优化)
```

## 模式 6: 内容更新

**症状**: 引用了废弃的 API 或工具

**修复**:
1. 识别过时的引用
2. 更新为当前版本
3. 更新 Maintenance 章节

## 模式选择决策树

```
SKILL.md > 200 行？
  ├─ 是 → 模式 1: 内容精简
  └─ 否 → 触发不可靠？
      ├─ 是 → 模式 2: 触发优化
      └─ 否 → 输出不一致？
          ├─ 是 → 模式 3: 指令冲突消除
          └─ 否 → 示例不足？
              ├─ 是 → 模式 4: 示例补充
              └─ 否 → 范围太广？
                  ├─ 是 → 模式 5: 范围拆分
                  └─ 否 → 内容过时？
                      ├─ 是 → 模式 6: 内容更新
                      └─ 否 → 可能不需要修改
```
