# AI ↔ 人类交互协议

定义架构可视化场景下 AI 与人类之间的标准化交互格式。

## 请求格式

### 标准请求模板

人类向 AI 发起架构可视化请求时，建议使用以下格式：

```markdown
## 架构可视化请求

### 目标
{一句话说明你要什么}

### 输出格式
- [ ] Mermaid flowchart（架构图）
- [ ] Mermaid sequenceDiagram（时序图）
- [ ] Canvas JSON（Obsidian 白板）
- [ ] 文字描述

### 粒度级别
- [ ] A-文件级（每个文件一个节点）
- [ ] B-类/函数级（每个类一个节点）
- [ ] C-服务级（只显示大模块）
- [ ] D-系统级（跨仓库/跨服务）

### 关注点（可多选）
- [ ] 依赖关系
- [ ] 数据流
- [ ] 调用链
- [ ] 潜在风险
- [ ] 性能热点

### 补充信息
{可选：已知的架构信息、特别关注的模块等}
```

### 简化请求

对于简单场景，可以使用简化格式：

```
生成 {项目/模块} 的 {Mermaid/Canvas} 架构图，{粒度级别}，关注 {关注点}
```

**示例**：
- "生成这个项目的 Mermaid 架构图，服务级，关注依赖关系"
- "画出用户登录的时序图"
- "生成 Canvas 白板，文件级，标注风险点"

## 响应格式

### 标准响应模板

AI 输出架构可视化时，必须使用以下格式：

```markdown
## 📊 架构图

\`\`\`mermaid
{图表代码}
\`\`\`

## 🔍 洞察摘要

| 维度 | 内容 |
|:---|:---|
| **架构模式** | {识别的模式，如 MVC/微服务/单体} |
| **核心组件** | {N} 个主要节点 |
| **关键路径** | {最重要的调用链描述} |
| **依赖复杂度** | Low/Medium/High |
| **潜在风险** | {循环依赖/热点/技术债，无则写"未发现"} |

## ⚠️ 需要确认（如有）

1. {问题1：某个依赖关系是否正确？}
2. {问题2：是否需要展开某个模块？}

## 📝 下一步建议

- {建议1：如"建议拆分 XXX 模块"}
- {建议2：如"建议添加缓存层"}
```

### 最小响应

对于简单请求，可以省略"需要确认"和"下一步建议"：

```markdown
## 📊 架构图

\`\`\`mermaid
{图表代码}
\`\`\`

## 🔍 洞察摘要

- **架构模式**: {模式}
- **核心组件**: {N} 个
- **关键路径**: {描述}
```

## 确认协议

### 何时需要人类确认

| 场景 | 行为 |
|:---|:---|
| 生成架构图 | 直接输出，无需确认 |
| 修改代码前 | **必须确认** |
| 不确定的依赖关系 | 在"需要确认"中列出 |
| 发现潜在风险 | 在"洞察摘要"中标注 |
| 多种可行方案 | 列出选项，请求选择 |

### 确认请求格式

```markdown
## ⚠️ 需要您确认

在继续之前，请确认以下内容：

1. **{问题1}**
   - 选项 A: {描述}
   - 选项 B: {描述}
   - 我的建议: {推荐选项及理由}

2. **{问题2}**
   - {描述问题}
   - 需要您提供: {所需信息}

请回复确认或提供补充信息。
```

## 迭代协议

### 增量更新

当人类要求修改已生成的架构图时：

```markdown
## 📊 更新后的架构图

\`\`\`mermaid
{更新后的图表代码}
\`\`\`

## 🔄 本次变更

| 变更类型 | 内容 |
|:---|:---|
| 新增 | {新增的节点/连线} |
| 修改 | {修改的内容} |
| 删除 | {删除的内容} |

## 🔍 更新后洞察

{更新后的摘要，特别是变更影响}
```

### 版本对比

当需要对比多个版本时：

```markdown
## 📊 版本对比

### 版本 A（当前）
\`\`\`mermaid
{当前版本}
\`\`\`

### 版本 B（修改后）
\`\`\`mermaid
{修改后版本}
\`\`\`

### 差异说明

| 方面 | 版本 A | 版本 B |
|:---|:---|:---|
| 模块数 | {N} | {M} |
| 依赖数 | {N} | {M} |
| 复杂度 | {评估} | {评估} |

### 建议

{选择哪个版本的建议及理由}
```

## 错误处理

### 信息不足

```markdown
## ❓ 需要更多信息

无法生成完整的架构图，因为缺少以下信息：

1. **{缺失信息1}**
   - 影响: {无法确定的部分}
   - 请提供: {具体需要什么}

2. **{缺失信息2}**
   - 影响: {无法确定的部分}
   - 请提供: {具体需要什么}

### 临时方案

基于现有信息，我可以生成一个简化版本：

\`\`\`mermaid
{简化版图表}
\`\`\`

**注意**: 此图可能不完整，标记为 `?` 的部分需要确认。
```

### 检测到问题

```markdown
## ⚠️ 检测到架构问题

在分析过程中发现以下问题：

### 1. {问题类型}（严重程度: High/Medium/Low）

**问题描述**: {描述}

**位置**: {涉及的模块/文件}

**建议**: {修复建议}

---

是否需要我在架构图中标注这些问题？
```

## 快捷指令

为提高效率，支持以下快捷指令：

| 指令 | 含义 |
|:---|:---|
| `/arch` | 生成 Mermaid 架构图 |
| `/seq` | 生成时序图 |
| `/canvas` | 生成 Canvas JSON |
| `/refresh` | 重新分析并更新架构图 |
| `/expand {模块}` | 展开指定模块的细节 |
| `/simplify` | 简化当前图（提高抽象级别） |
| `/risks` | 标注潜在风险点 |
