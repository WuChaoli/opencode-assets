# Task Breakdown Guide

## 原则

### 1. 任务粒度

每个 task 应该：
- 能在一个 subagent 会话中完成（不超过 30 分钟的工作量）
- 有明确的输入和输出
- 有可验证的完成标准
- 不跨多个技术栈（前端 task 只做前端，后端 task 只做后端）

### 2. 依赖管理

- 识别 task 之间的依赖关系
- 无依赖的 task 可以并行执行
- 有依赖的 task 必须串行执行
- 优先执行被依赖最多的 task（关键路径优先）

### 3. 任务分类

按技术栈分类：
- **frontend**: UI 组件、页面、样式、状态管理
- **backend**: API、服务层、数据模型、中间件
- **android**: Activity、Fragment、ViewModel、UI
- **test**: 单元测试、集成测试、E2E 测试
- **analyze**: 代码探索、技术调研、方案设计

## Task 拆解流程

### Step 1: 理解 PRD

1. 提取所有功能点
2. 识别技术约束
3. 确定非功能需求（性能、安全、兼容性）

### Step 2: 模块划分

1. 按功能域划分模块
2. 确定模块间的依赖关系
3. 确定模块的技术栈

### Step 3: 任务拆解

对每个模块：
1. 拆解为技术组件
2. 每个组件拆解为具体 task
3. 标注 task 类型、优先级、依赖

### Step 4: 验证

1. 检查是否覆盖 PRD 的所有需求
2. 检查依赖关系是否完整
3. 检查 task 粒度是否合适
4. 检查是否有重复或遗漏

## Task 模板

```markdown
### Task: {task-name}
- **描述**: {一句话描述}
- **类型**: {frontend|backend|android|test|analyze}
- **优先级**: {P0|P1|P2}
- **依赖**: [{task-name-1}, {task-name-2}]
- **分配给**: {subagent-name}
- **输入**: 
  - {输入 1}
  - {输入 2}
- **输出**:
  - {输出 1}
  - {输出 2}
- **验证标准**:
  - [ ] {验证点 1}
  - [ ] {验证点 2}
- **备注**: {额外说明}
```

## 示例

### 示例 1: 用户登录功能

```markdown
### Task: user-login-api
- **描述**: 实现用户登录 API，支持用户名密码和 OAuth2
- **类型**: backend
- **优先级**: P0
- **依赖**: [database-setup]
- **分配给**: backend-dev
- **输入**: 
  - 用户数据模型
  - OAuth2 配置
- **输出**:
  - POST /api/auth/login 端点
  - JWT token 生成逻辑
- **验证标准**:
  - [ ] 登录成功返回 JWT token
  - [ ] 登录失败返回明确错误信息
  - [ ] 密码加密存储
  - [ ] 单元测试通过

### Task: user-login-ui
- **描述**: 实现登录页面 UI
- **类型**: frontend
- **优先级**: P0
- **依赖**: [user-login-api]
- **分配给**: frontend-dev
- **输入**: 
  - 设计稿
  - API 接口文档
- **输出**:
  - 登录页面组件
  - 表单验证逻辑
- **验证标准**:
  - [ ] 表单验证正常工作
  - [ ] 登录成功后跳转
  - [ ] 错误提示友好
  - [ ] 响应式布局
```

## 常见错误

| 错误 | 症状 | 修复 |
|------|------|------|
| 任务过大 | subagent 执行超时或上下文溢出 | 拆分为更小的 task |
| 依赖缺失 | task 执行时发现缺少前置条件 | 补充依赖关系 |
| 类型错误 | 前端 task 包含了后端逻辑 | 重新分类 task |
| 验证模糊 | 无法判断 task 是否完成 | 明确验证标准 |
| 重复任务 | 多个 task 做同一件事 | 合并或重新划分 |
