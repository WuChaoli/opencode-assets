# 常见模式与反模式

## 好 vs 坏对比

### 1. description 写法

❌ 坏: `"帮助开发者写更好的代码"`（太模糊，无法判定触发）
✅ 好: `"胶水开发模式：强依赖复用成熟库。触发条件：新功能开发、代码生成时。"`

### 2. Quick Reference 写法

❌ 坏: 大段解释文字，需要阅读才能理解
✅ 好: 可直接复制的命令/代码片段，一行说明

### 3. Examples 写法

❌ 坏: 只有输入和输出，没有步骤
✅ 好: 输入 → 步骤 → 验收标准，可复现

### 4. 边界定义

❌ 坏: 没有 Not For 章节
✅ 好: 明确列出不适用的场景和必要输入

## 常见错误

| 错误 | 症状 | 修复 |
|------|------|------|
| 触发不可靠 | 有时触发有时不触发 | description 加入具体关键词 |
| 内容太长 | SKILL.md 超过 300 行 | 长内容移到 references/ |
| 示例不可复现 | 按示例操作得不到预期结果 | 补充前提条件，实际运行验证 |
| 凭空发明 | 资料中没有的内容被写入 | 标注"文档未提及，需验证" |
| 跳过确认 | 用户不满意创建结果 | 严格执行 8 步流程，先确认后创建 |
| 闭门造车 | 忽略网上可复用经验 | 创建前先调研，整理调研报告 |
